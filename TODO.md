# TAAC 2026 冠军导向技术路线图

## 总目标

围绕比赛主题“统一序列建模与特征交互”，构建一个单模型、统一骨干、候选感知、满足推理延迟约束的 CTR/CVR 预测系统。

最终方案必须同时满足以下四个目标：

1. 排行榜指标强，主指标 AUC 持续提升。
2. 架构统一，不依赖多塔拼装和模型集成。
3. 推理高效，天然适配严格时延限制。
4. 技术叙事完整，可支撑创新奖申报与论文撰写。

## 核心判断

这道题的最优解不太可能是传统特征交互模型与序列模型的简单拼接，也不太可能是朴素大 Transformer 的直接套用。更可行的方向是：

1. 用统一 token 化方式同时表达用户静态特征、上下文特征、候选 item 特征、历史行为事件。
2. 用同质化骨干网络处理异构 token，在每层中持续建模候选 item 与历史行为、多字段特征之间的交互。
3. 用分层建模和压缩建模解决长序列开销，而不是依赖无约束全注意力。
4. 从第一版开始就把时延和可扩展性作为一等约束，而不是后期补救。

## 主线方案

### 方案名称

Unified Candidate-Aware Sequence Interaction Model

简称 UCASIM。

### 方案目标

以单一统一骨干，打通以下两类信息：

1. 用户历史行为序列中的时序动态。
2. 用户、候选 item、上下文、多字段 sparse feature 之间的高阶交互。

### 输入表示

所有输入被转换为统一 token 序列，但保留类型信息。

输入由四部分组成：

1. 用户静态特征 token
2. 上下文特征 token
3. 候选 item 特征 token
4. 历史行为事件 token

每类 token 统一映射为 d 维表示，并显式注入：

1. feature_id embedding
2. feature_type embedding
3. value embedding 或 value projection
4. field group embedding
5. source embedding，用于区分 user、context、target item、history event

### 历史事件建模

每个历史事件不是单一 item_id，而是一个事件级复合表示，包含：

1. 历史 item identity
2. action_type
3. event timestamp 或 time bucket
4. 相对当前样本时间的 time gap
5. 相对位置 position
6. 可选的事件侧属性特征

为了增强候选感知，事件表示中引入在线构造的候选相关特征，例如：

1. 历史 item 与候选 item 的相同类目标记
2. 历史 item 与候选 item 的 shared feature overlap
3. 历史行为类型与当前预测任务的相关性标记
4. 基于时间窗口的近期偏好匹配特征

### 主干结构

统一骨干由三类 block 组成：

1. Local Interaction Block
2. Global Compression Block
3. Candidate Readout Block

#### Local Interaction Block

负责建模最近一段高信息密度行为与候选 item 的细粒度交互。

建议设计：

1. 保留最近 N_short 个历史事件。
2. 对这部分事件与候选 token 做全交互或局部窗口交互。
3. 静态特征和上下文 token 通过轻量 cross-attention 或门控调制参与融合。

目标是捕捉短期兴趣、最近曝光点击链路、局部时序模式。

#### Global Compression Block

负责对长历史进行低成本建模，避免全量 self-attention。

建议设计：

1. 对较远历史按时间窗口、行为类型或聚类方式做 segment。
2. 每个 segment 压缩为少量 memory token。
3. 用 memory token 与候选 token 进行交互，而不是保留全部远程事件。

候选压缩机制候选方案：

1. 时间分桶压缩
2. 行为类型分桶压缩
3. top-k 相关事件选择
4. 基于当前候选的 attention pooling

目标是保留长时偏好与稀有高价值行为，而将复杂度控制在可提交范围内。

#### Candidate Readout Block

用候选 token 作为持续读出中心，而不是简单池化后接 MLP。

具体原则：

1. 每层都允许候选 token 主动读取历史和多字段特征。
2. 输出阶段以候选相关表示为主，不采用用户塔和 item 塔独立编码后晚融合。
3. 最终 logit 建立在 candidate-aware 表示之上。

### 输出头设计

主任务是比赛目标对应的二分类输出。

如果标签包含多种 action_type，可训练时增加辅助任务：

1. 多行为预测
2. 未来时间窗行为预测
3. 序列一致性或 next-event 辅助损失

但提交模型保持单模型单输出头，避免复杂部署链路。

## 数据工程路线

### 数据层原则

先做一套稳定、可缓存、可扩展的数据处理框架，而不是把特征处理硬编码在训练脚本里。

### 数据处理目标

1. 支持嵌套结构样本解析。
2. 支持多 feature_value_type 的统一编码。
3. 支持离线缓存与快速重跑。
4. 支持时间切分验证。
5. 支持统计特征和动态序列特征并行构造。

### 样本拆解策略

针对每条样本拆成如下子结构：

1. target item block
2. user feature block
3. context block
4. sequence event block
5. label block

### 特征编码策略

稀疏离散特征：

1. 统一词表或哈希映射
2. 按 field 共享或半共享 embedding
3. 高频 field 保留独立 embedding table

数值特征：

1. 分桶 embedding
2. 标准化后线性投影
3. 对长 float_array 使用小型 MLP 或线性投影降维

序列特征：

1. 保留原始顺序
2. 增加时间差编码
3. 保留行为类型与位置编码

### 验证集构造

验证集优先采用时间切分，而不是随机切分。

理由：

1. 推荐场景天然存在时序漂移。
2. 随机切分会高估模型效果。
3. 时间切分更接近线上评测分布。

建议同时维护两套验证：

1. 快速随机切分，用于排错和快速 smoke test
2. 正式时间切分，用于模型选择

## 实验路线

### 阶段一：建立强基线

目标是在最短时间内建立可复现、可扩展、可对比的实验闭环。

交付物：

1. 数据读取与缓存管线
2. 统一 token 编码器
3. 可训练 baseline 模型
4. AUC 评估脚本
5. 配置系统与实验记录

首个 baseline 不追求最强，只要求：

1. 单卡或多卡稳定训练
2. 结果可复现
3. 模型结构便于后续替换 block

建议 baseline：

1. 最近历史事件截断
2. 简化候选感知 attention
3. 不引入复杂压缩模块

### 阶段二：围绕三条主假设做提升

只重点验证最有希望改变上限的三类改动：

1. 候选感知历史建模
2. 长历史压缩机制
3. 多字段特征注入位置

每类假设都应以成组消融推进。

#### 假设 A：候选感知比纯用户序列表征更重要

对比实验：

1. 历史独立编码后晚融合
2. 候选 token 每层读取历史
3. 仅最终层 candidate cross-attention
4. 全层 candidate-aware interaction

#### 假设 B：长序列需要压缩而不是盲目拉长

对比实验：

1. 固定截断长度
2. 最近窗口加远程压缩 memory
3. top-k 相关事件选择
4. 时间窗口分段压缩

#### 假设 C：静态特征的注入位置会影响统一建模收益

对比实验：

1. 输入层直接拼接
2. 中间层门控调制
3. 多层 cross-attention 注入
4. 仅输出层融合

### 阶段三：面向提交的系统优化

这阶段的目标不再是盲目提 AUC，而是在几乎不损失 AUC 的前提下把延迟打下来。

重点优化方向：

1. token pruning
2. sequence truncation policy
3. memory token 数量压缩
4. embedding table 规模控制
5. block 数与 hidden size 联合搜索
6. mixed precision 与 fused operator 使用

## 模型搜索优先级

### 优先做

1. 候选感知 attention 结构
2. 事件表示中的时间差编码
3. 长序列压缩模块
4. user/context/item feature 的统一编码接口
5. 与时延相关的 token 数控制

### 暂缓做

1. 复杂 MoE
2. 大规模蒸馏链路
3. 多模型融合
4. 纯论文导向但部署复杂的 exotic 模块

原因是这些方向工程成本高，且不一定适配比赛规则。

## 消融实验清单

每轮实验至少记录以下维度：

1. AUC
2. 参数量
3. token 总数
4. 单样本推理延迟
5. 吞吐量
6. 显存占用
7. 训练稳定性

优先消融：

1. 历史长度 20、50、100、200
2. 是否加入 time gap encoding
3. 是否加入 action_type embedding
4. 候选感知交互层数
5. memory token 数量
6. 不同 feature group 的注入方式

## 指标体系

公开主指标是 AUC，但内部必须维护多维指标。

建议看板：

1. 主验证 AUC
2. 不同行为频次用户分桶 AUC
3. 不同序列长度分桶 AUC
4. 不同时间窗口分桶 AUC
5. 推理 P50、P95 延迟
6. 每秒样本吞吐

如果没有分桶评估，很容易出现整体 AUC 提升但关键长序列场景退化的问题。

## 风险点

### 风险 1：样例数据太小导致结构判断失真

公开样例只有 1000 条，不能据此对复杂架构做强结论。

应对：

1. 先把框架搭好
2. 小样本只做功能验证
3. 正式数据到位后再做结构优选

### 风险 2：统一建模带来 token 爆炸

如果把所有 feature 都变成 token，复杂度可能不可接受。

应对：

1. 区分核心 token 与辅助 token
2. 对低价值字段做 pooling 或压缩
3. 对 float_array 类特征先投影后注入

### 风险 3：延迟限制后期卡死

如果一开始忽视延迟，后期往往要推翻重来。

应对：

1. 训练阶段就记录延迟
2. 每个版本同步评估参数量和 token 数
3. 主干设计优先选择结构上可裁剪的模块

### 风险 4：实验空间过大，试验无序

应对：

1. 只围绕三个主假设推进
2. 采用固定模板记录实验
3. 每周只保留少数重点分支

## 冠军级实施节奏

### 第 1 周

1. 完成数据解析与 schema 抽象
2. 完成最小训练闭环
3. 搭建时间切分验证
4. 产出第一个 unified baseline

### 第 2 周

1. 引入 candidate-aware interaction
2. 跑第一轮关键消融
3. 确定事件表示方案
4. 确定历史长度基线

### 第 3 周

1. 引入长序列压缩模块
2. 建立延迟与 AUC 联合评估表
3. 固化当前最优主线

### 第 4 周及以后

1. 围绕主线结构做精细搜索
2. 开始写技术报告骨架
3. 同步整理 scaling law 实验
4. 为创新奖准备方法论叙事

## 创新奖叙事建议

如果目标不仅是冲榜，还要冲创新奖，建议围绕两个主张展开。

### 统一模块创新奖

主张：

1. 用统一 token 化和统一骨干，将序列建模与多字段特征交互合并到一个 stack 内。
2. 用候选感知读出机制替代传统双塔或晚融合设计。

### 缩放规律创新奖

主张：

1. 分析模型容量、历史长度、memory token 数、特征 token 数与 AUC 的关系。
2. 给出时延约束下的最优 scaling 区间，而不是只报单点最好结果。

## 近期立即执行事项

按优先级排序：

1. 建立项目目录结构，拆分 data、model、train、config、eval 模块。
2. 实现 parquet 样本读取与嵌套结构解析。
3. 抽象统一特征编码接口，支持 int_value、int_array、float_array。
4. 写第一个 candidate-aware baseline。
5. 接入 AUC 与时间切分评估。
6. 加入最基础的推理延迟测量脚本。

## 结论

主线不应是“做一个尽可能大的模型”，而应是“做一个单模型、统一、候选感知、长序列可压缩、时延友好”的系统。

如果这条路线走通，这个方案既有冲排行榜的现实可能，也有写出有辨识度技术报告的空间。
